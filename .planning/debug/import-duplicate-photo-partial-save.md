## Symptoms

- expected: Adding the same photo twice (or overlapping photos across finds in one import batch) should either be silently deduplicated so Save succeeds cleanly, or Save should cleanly fail with a clear message — it must never partially succeed while destroying source files.
- actual: When a duplicate photo is present in an import batch, Save sometimes fails, but the source file(s) are still deleted from the source folder (deleteSource defaults to true), and the find still ends up saved into the app — in an incomplete/inconsistent state (missing photos, or a corrupted assumption about what was imported).
- errors: Not yet systematically captured — user has not pasted an exact error message. Needs reproduction to capture the exact Rust/frontend error surfaced.
- timeline: Ongoing, not described as a fresh regression.
- reproduction: In "Uvezi fotografije" (Import photos) dialog, add the same photo twice (or add overlapping photos across two finds in the same import batch), enable "Ukloni izvornu datoteku nakon uvoza" (delete source, default ON), and click Save/Import.

## User's Preferred Fix Priority (from their own comment, most to least preferred)

1. Best: recognize identical photos and silently keep only one so Save never errors for this reason.
2. If not reliably feasible in all cases: fully block Save with a clear message instead of a partial/half success.

## Hypotheses

1. PRIMARY: `import_find`'s lack of a SQL transaction + per-item immediate delete-after-copy means a failure partway through a multi-payload batch leaves earlier items durably committed (finds+photos in DB, source files deleted) while the overall call reports failure — a genuine non-atomicity bug, independent of what specifically triggers the mid-batch `std::fs::copy` error.
2. SECONDARY: There is a frontend path (multi-find bulk grouping, or a photo-path edge case not caught by `dedupePhotoPaths`'s path-string normalization) that lets a duplicate/colliding photo reach the Rust layer as two payloads that BOTH attempt to write into the same destination sequence slot, causing the copy step to fail. Needs confirmation before treating as certain.

## Resolution

- root_cause: >
    `import_find` (src-tauri/src/commands/import.rs) performed its find-row insert, primary-photo copy+delete+insert, and each additional-photo copy+delete+insert as a sequence of independently-autocommitted SQLite writes and destructive filesystem operations, with no surrounding transaction and no rollback/compensation on failure. Because `std::fs::copy(...)? ` used early-return (`?`) on error, a failure on ANY step after the first successful copy+delete (most commonly: copying an additional photo whose source file is a duplicate-by-path of one already consumed/deleted earlier in the same or a prior retried call, or any transient copy failure such as a locked/moved/removed file) aborted the whole command with `Err(...)`. Everything already durably written before that point (the `finds` row, `find_photos` rows for the primary and any already-processed additional photos, with their SOURCE FILES ALREADY DELETED if delete_source=true) remained permanently committed in SQLite even though the command reported total failure to the frontend. The frontend's error handler (ImportDialog.tsx handleImportAll catch block) does not reset `photos`/form state, so a natural user retry re-submits the same (now partially-consumed) source paths, compounding the problem. This precisely reproduced the reported symptom of "Save fails, source file(s) deleted anyway, find ends up saved in an incomplete state."
  fix: >
    Restructured `import_find` in src-tauri/src/commands/import.rs into three explicit phases per payload: (1) COPY phase (`copy_payload_photos`) — copies every photo (primary + all additional_photos) into storage first, with NO database writes and NO source deletion. If any copy fails partway, all destination files already copied for that find are removed (`cleanup_staged_photos`, only for freshly-copied files, never for `is_already_in_storage` in-place files) and the source files are left completely untouched, then the error propagates. (2) COMMIT phase — only after every copy for the find succeeds, a single rusqlite transaction (`conn.transaction()` / `tx.commit()`, mirroring the existing `update_find` pattern) inserts the find row, the species common-name upsert, and every photo row; if this fails, the transaction is dropped (rusqlite rolls back automatically) and the already-copied destination files are cleaned up, again without touching any source file. (3) DELETE phase — only after the transaction commits successfully does source deletion run (best-effort, retried, as before), and only for photos that were actually freshly copied (never for in-place auto-import photos). This makes a durably-half-imported find impossible: either the whole find (all photos) commits, or nothing does, and source files are only ever deleted after that commit succeeds -- satisfying fix priority #3 (never delete source files that were not durably imported) and priority #2 (fail cleanly/atomically instead of partially).

    FOLLOW-UP FIX (2026-07-06): `is_likely_duplicate_content` (src-tauri/src/commands/import.rs, ~line 704) originally required an exact case-insensitive filename match as a precondition before even checking file size — meaning it could never catch the "same photo copied under a different filename" case (e.g. Windows' auto-generated " - Copy" suffix) despite its own doc comment and this session's own suggested verification step explicitly claiming it does. Fixed by removing the filename-equality precondition entirely: the function now checks file size first via `std::fs::metadata` (cheap, keeps the common case of differently-sized non-duplicate photos fast), and only when sizes match does it read both files fully via `std::fs::read` and compare bytes for exact equality. The doc comment was rewritten to accurately describe this behavior and explicitly calls out that filename is no longer part of the check. No other part of `copy_payload_photos`/`import_find` was touched — the call site and function signature are unchanged, so this is a self-contained fix to the helper's internal logic, correctly completing fix priority #1 (silently keep only one copy of identical photos, regardless of what they are named).
