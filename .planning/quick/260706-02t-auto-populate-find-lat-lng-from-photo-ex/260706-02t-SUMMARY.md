# Summary: Auto-populate find lat/lng from photo EXIF GPS during import

## What shipped

- `src-tauri/src/commands/import.rs`:
  - `first_gps_coords_from_paths(paths: &[&str]) -> Option<(f64, f64)>` — scans photo source paths in order, returns the first GPS coordinate pair found via `extract_exif`, `None` if none have GPS.
  - `first_gps_coords_from_staged(staged: &[StagedPhoto])` — thin wrapper over the above for the staged-photo list used during import.
  - `resolve_find_coords(payload_lat, payload_lng, exif_coords) -> (Option<f64>, Option<f64>)` — manual payload coordinates always win; EXIF fallback only applies when the payload supplied neither lat nor lng.
  - `import_find` now computes `(final_lat, final_lng)` via `resolve_find_coords` right before constructing `FindRecord`, using `first_gps_coords_from_staged(&staged)` (staged photos already reflect primary + additional_photos in insertion order).
  - Added unit tests for `resolve_find_coords` (4 cases), `first_gps_coords_from_paths` (no-GPS, empty slice), and `first_gps_coords_from_staged` (no-GPS staged photo).
- `src-tauri/src/commands/finds.rs`:
  - `add_find_photos` now backfills a find's lat/lng after the photo copy+insert loop, via `UPDATE finds SET lat = ?1, lng = ?2 WHERE id = ?3 AND lat IS NULL AND lng IS NULL`, using coordinates from `first_gps_coords_from_paths` (imported from `import.rs`) over the newly added `source_paths`. The `WHERE ... IS NULL` guard makes this a no-op whenever the find already has coordinates.
  - Added tests: backfill sets lat/lng when both null, guarded UPDATE never touches an already-set find (0 rows affected + values verified intact), guard behavior for both null/set cases via direct row-count assertions, and a cross-module reachability test for `first_gps_coords_from_paths`.
- `EditFindDialog.tsx` / `LocationPickerMap` / `CollectionPins.tsx`: unchanged — manual lat/lng entry and the map picker continue to work exactly as before, and now simply overwrite whatever EXIF may have auto-populated.

## Process deviation (documented for traceability)

Plan execution was first delegated to a `gsd-executor` running in an isolated git worktree. That worktree was created from the last **committed** state of `import.rs` (`d6f0148`), which does not include the in-progress, uncommitted copy/commit/delete atomicity refactor that exists only in this repo's main working tree. The executor correctly adapted to the interfaces it could actually see there (scanning `payload.source_path` + `payload.additional_photos` directly) and produced two clean commits on its branch.

Because the target files (`import.rs` especially) were heavily modified but uncommitted on `main`, merging that worktree branch back with a plain `git merge` would have collided with the real, in-progress refactor. Instead, the same logic (identical helper functions and wiring, adapted to the actual current `StagedPhoto`/`copy_payload_photos` structure present in `import.rs`) was manually re-applied directly on top of the current working tree in the orchestrating session, and `finds.rs` (which had no local uncommitted changes) was taken as-is from the executor's branch via `git checkout <branch> -- src-tauri/src/commands/finds.rs`. The executor's worktree and branch were then removed.

One consequence: the executor's own `260706-02t-SUMMARY.md` (written inside its worktree) was not committed by design (docs artifacts are committed by the orchestrator, not the executor) and was lost when the worktree was force-removed before extracting it. This file reconstructs that summary from the executor's final report plus the actual changes verified on `main`.
