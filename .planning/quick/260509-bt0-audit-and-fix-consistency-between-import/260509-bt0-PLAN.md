<objective>
Fix consistency issues between import flow and edit flow for shared find/species metadata.

Purpose: After auditing all three find creation paths (ImportDialog, CreateFindDialog, EditFindDialog) and the species-level editor (FolderEditDialog), three real inconsistencies were found:

1. ImportDialog conflates species folder notes and find-level notes into one textarea, saving the same text as both `notes` on the find record AND via `upsertSpeciesNote`. Find-level notes should be separate from species description.

2. CreateFindDialog has no edibility/protected status fields, while ImportDialog and FolderEditDialog both offer them. When creating a new species via "New Find", you cannot set species metadata.

3. FindPreviewCard.tsx and its tests are dead code -- ImportDialog was redesigned to use a single-find flow with inline photo thumbnails. The old per-card preview with its outdated single `observed_count` field (no range) is never rendered.

Output: Cleaned-up import/create flows with consistent field semantics.
</objective>

  <files>src/components/import/ImportDialog.tsx</files>
  <action>
In ImportDialog.tsx, the `sharedFolderNotes` state is currently used for BOTH:
- The find record's `notes` field (passed as `payload.notes` in the ImportPayload)
- The species notes (passed to `upsertSpeciesNote`)

This is wrong: the find's `notes` field should hold per-find observations (e.g. "Found near old oak, very fresh"), while species notes are a general description of the species.

Fix:
1. Keep `sharedFolderNotes` state + textarea for species-level notes (label: `t('import.folderNotes')` -- already correct).
2. Add a new `sharedFindNotes` state (default: empty string).
3. Add a new Textarea BELOW the species notes textarea, labeled "Find notes" / per-find observations. Use placeholder like "Notes about this specific find..." or the i18n equivalent. Use same styling as the species notes textarea (rows=2).
4. In `handleImportAll`, change the payload to use `sharedFindNotes` for `notes`:
   ```
   notes: sharedFindNotes.trim(),
   ```
   Keep the `upsertSpeciesNote` call using `sharedFolderNotes` unchanged.
5. Add `sharedFindNotes` to `resetState()` (reset to empty string).
6. Add reset in the state declarations alongside the other `shared*` states.

This ensures species notes and find notes are semantically separate, matching how EditFindDialog (find notes only) and FolderEditDialog (species metadata only) work.

<task type="auto">
  <name>Task 2: Add edibility and protected status to CreateFindDialog</name>
  <files>src/components/finds/CreateFindDialog.tsx</files>
  <action>
CreateFindDialog currently has no species metadata fields (edibility, protected status). ImportDialog and FolderEditDialog both offer them. Add consistency:

1. Import species metadata types and hooks:
   ```ts
   import { EDIBILITY_VALUES, EDIBILITY_LABELS, PROTECTED_STATUS_VALUES, PROTECTED_STATUS_LABELS } from '@/lib/speciesMetadata';
   import { useSpeciesProfiles, useUpsertSpeciesProfile } from '@/hooks/useFinds';
   ```

2. Add state for edibility and protected status:
   ```ts
   const [edibility, setEdibility] = useState<string>('unknown');
   const [protectedStatus, setProtectedStatus] = useState<string>('unknown');
   ```

3. Add hooks:
   ```ts
   const { data: speciesProfilesData } = useSpeciesProfiles();
   const upsertProfile = useUpsertSpeciesProfile();
   ```

4. Add a useEffect that pre-fills edibility/protected from existing species profile when species name changes (same pattern as ImportDialog):
   ```ts
   useEffect(() => {
     if (!speciesProfilesData || !form.species_name) return;
     const existing = speciesProfilesData.find(
       (p) => p.species_name.toLowerCase() === form.species_name.toLowerCase(),
     );
     setEdibility(existing?.edibility ?? 'unknown');
     setProtectedStatus(existing?.protected_status ?? 'unknown');
   }, [form.species_name, speciesProfilesData]);
   ```

5. Add the two `<select>` dropdowns AFTER the observed count field, using the same layout as FolderEditDialog (grid-cols-2 gap-2):
   ```tsx
   <div className="grid grid-cols-2 gap-2">
     <div>
       <label className="text-sm font-medium">Edibility</label>
       <select value={edibility} onChange={(e) => setEdibility(e.target.value)}
         className="mt-1 w-full h-9 rounded-md border border-border bg-input px-3 text-sm text-foreground focus:outline-none focus:ring-2 focus:ring-ring/40">
         {EDIBILITY_VALUES.map((v) => (
           <option key={v} value={v}>{EDIBILITY_LABELS[v]}</option>
         ))}
       </select>
     </div>
     <div>
       <label className="text-sm font-medium">Protected Status</label>
       <select value={protectedStatus} onChange={(e) => setProtectedStatus(e.target.value)}
         className="mt-1 w-full h-9 rounded-md border border-border bg-input px-3 text-sm text-foreground focus:outline-none focus:ring-2 focus:ring-ring/40">
         {PROTECTED_STATUS_VALUES.map((v) => (
           <option key={v} value={v}>{PROTECTED_STATUS_LABELS[v]}</option>
         ))}
       </select>
     </div>
   </div>
   ```

6. In `handleSave`, after `createMutation.mutate` succeeds, upsert the species profile if either field is not 'unknown'. Use the same pattern as ImportDialog:
   ```ts
   createMutation.mutate(
     { /* existing payload */ },
     {
       onSuccess: () => {
         if (edibility !== 'unknown' || protectedStatus !== 'unknown') {
           const existing = speciesProfilesData?.find(
             (p) => p.species_name === form.species_name.trim(),
           );
           upsertProfile.mutate({
             speciesName: form.species_name.trim(),
             coverPhotoId: existing?.cover_photo_id ?? null,
             tags: existing?.tags ?? [],
             edibility: edibility === 'unknown' ? null : edibility,
             protectedStatus: protectedStatus === 'unknown' ? null : protectedStatus,
           });
         }
         onOpenChange(false);
       },
     },
   );
   ```

7. Reset edibility and protectedStatus to 'unknown' in the dialog open useEffect.
