<decisions>
## Implementation Decisions

### 1. Schema Migration (Breaking Change)
Remove `photo_path TEXT` from `finds` table. Add a `find_photos` table:
```sql
CREATE TABLE find_photos (
  id INTEGER PRIMARY KEY,
  find_id INTEGER NOT NULL REFERENCES finds(id) ON DELETE CASCADE,
  photo_path TEXT NOT NULL,
  is_primary INTEGER NOT NULL DEFAULT 0,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```
Migration file: `src-tauri/migrations/0003_find_photos.sql`. The `is_primary = 1` row is the card thumbnail. All existing `finds.photo_path` values must be migrated into `find_photos` rows as primary photos in migration 0003.

### 2. Import Modes
Two modes, selected via an upfront toggle at the top of ImportDialog (before picking files):

**Mode A — "One mushroom" (default):** All selected photos belong to the same find. A single find record is created with N photos attached. All preview cards share the same species name, date, country, region. Individual lat/lng can vary per card (useful if the user walked a bit between shots).

**Mode B — "Multiple mushrooms":** Each photo becomes its own find record (current behavior). Preview cards are independent.

Mode A is the default because most imports are "I took 5 shots of the same chanterelle".

### 3. Bulk Metadata
In both import modes, a "Apply to all cards" action exists at the top of the preview list. It lets the user fill in: Country, Region, Date, Species name (in Mode B), and optionally lat/lng — then push those values to every card at once. Individual cards can still override afterwards.

### 4. Map Location Picker
A "Set location for all" button at the top of ImportDialog opens a shared map modal (mini Leaflet map). User drops a single pin → lat/lng auto-fills for ALL preview cards simultaneously. Each preview card also has a "Pick on map" button for individual override. Both the shared and per-card map modal use the same `LocationPickerMap` component.

**Map scope:** Basic click-to-pin + OSM tile layer only. No offline caching (Phase 3). react-leaflet and leaflet are installed in this phase (pulled forward from Phase 3).

### 5. Post-Import Review Dialog
After clicking "Import All" and the import completes, a modal opens showing the just-imported finds with full edit-in-place (same fields as EditFindDialog). User can correct species name, date, location for any find. The dialog has a "Done" button to close. This covers the use-case of quickly verifying a batch import is correct.

### 6. Delete Find
FindCard gets a delete button (trash icon). Clicking it shows a confirmation dialog with two options:
- "Delete find only" — removes DB record(s), leaves photo file on disk
- "Delete find + photos" — removes DB record(s) and moves photo file(s) to the system trash/recycle bin via Tauri's `trash` plugin or `fs` plugin.

The Rust command `delete_find(find_id, delete_files: bool)` handles both paths.

### 7. FindCard Photo Strip
FindCard shows a horizontal strip of up to 4 thumbnails from `find_photos`. If a find has more than 4 photos, the 4th slot shows a badge "+ N more". The primary photo (is_primary = 1) is always the first thumbnail.

### 8. HEIC Photos
HEIC files continue to show the placeholder icon in thumbnails (no conversion in this phase). The HEIC placeholder shows as a single slot in the photo strip.

</decisions>
