## Summary

Phase 02.1 is a focused refactor of the existing import loop plus three new capabilities: multi-photo finds, map-based location picking, and delete-with-disk-option. The codebase is well-structured with clear separation of concerns ‚Äî heavy I/O in Rust Tauri commands, async state via TanStack Query, UI in React with shadcn/ui components.

The biggest risk is the schema migration. The `finds` table currently has a `photo_path TEXT NOT NULL` column that must be migrated into a new `find_photos` table. SQLite 3.35+ supports `DROP COLUMN` natively, but the bundled `libsqlite3-sys` version needs to be at least 3.35 (released March 2021); since the project already uses `features = ["bundled"]` with a recent `libsqlite3-sys`, this is safe. The safe migration pattern is: create `find_photos`, INSERT existing rows with `is_primary = 1`, then `ALTER TABLE finds DROP COLUMN photo_path`.

The location picker is straightforward using react-leaflet v5's `useMapEvents` hook, which provides a `click` handler giving access to `e.latlng`. No official Tauri trash plugin exists; the `trash` Rust crate (v5.2.5) is the correct dependency ‚Äî one call to `trash::delete(path)` moves a file to the Windows Recycle Bin.

All changes are additive to the existing test infrastructure (Vitest + jsdom + tauri-mocks dispatch table). New commands (`delete_find`, `get_find_photos`) follow the same mock-registration pattern already established in `tauri-mocks.ts`.

**Primary recommendation:** Work in four ordered areas: (1) Rust migration + new commands, (2) `LocationPickerMap` component, (3) `ImportDialog` refactor with mode toggle + bulk metadata, (4) `FindCard` + `CollectionTab` delete integration and photo strip.

### Pattern 1: LocationPickerMap (react-leaflet click-to-pin)

**What:** A modal containing a Leaflet `MapContainer`. A child component uses `useMapEvents` to intercept map clicks and place/move a `Marker`. A "Confirm" button writes the current pin coordinates back to the caller via a callback prop.

**When to use:** Both the shared "Set location for all" button in ImportDialog and the per-card "Pick on map" button in FindPreviewCard use this component.

```typescript
// Source: https://react-leaflet.js.org/docs/api-map/ (useMapEvents hook)
import { MapContainer, TileLayer, Marker, useMapEvents } from 'react-leaflet';
import type { LatLng } from 'leaflet';

interface LocationPickerMapProps {
  open: boolean;
  onOpenChange: (open: boolean) => void;
  initialLatLng?: { lat: number; lng: number } | null;
  onConfirm: (lat: number, lng: number) => void;
}

// Inner component that must be rendered inside MapContainer
function ClickHandler({ onPick }: { onPick: (latlng: LatLng) => void }) {
  useMapEvents({
    click(e) {
      onPick(e.latlng);
    },
  });
  return null;
}

export function LocationPickerMap({ open, onOpenChange, initialLatLng, onConfirm }: LocationPickerMapProps) {
  const [pin, setPin] = useState<LatLng | null>(
    initialLatLng ? L.latLng(initialLatLng.lat, initialLatLng.lng) : null
  );
  const center: [number, number] = initialLatLng
    ? [initialLatLng.lat, initialLatLng.lng]
    : [45.1, 15.2]; // Croatia region default

  return (
    <Dialog open={open} onOpenChange={onOpenChange}>
      <DialogContent className="max-w-2xl">
        <DialogHeader><DialogTitle>Pick location</DialogTitle></DialogHeader>
        <div style={{ height: 400 }}>
          <MapContainer center={center} zoom={initialLatLng ? 13 : 7} style={{ height: '100%', width: '100%' }}>
            <TileLayer url="https://{s}.tile.openstreetmap.org/{z}/{x}/{y}.png" />
            <ClickHandler onPick={setPin} />
            {pin && <Marker position={pin} />}
          </MapContainer>
        </div>
        <DialogFooter>
          <Button variant="outline" onClick={() => onOpenChange(false)}>Cancel</Button>
          <Button disabled={!pin} onClick={() => { if (pin) { onConfirm(pin.lat, pin.lng); onOpenChange(false); } }}>
            Confirm location
          </Button>
        </DialogFooter>
      </DialogContent>
    </Dialog>
  );
}
```

**Key constraint:** `useMapEvents` MUST be rendered as a child of `MapContainer`. It cannot be called in the parent component that renders `MapContainer`. The `ClickHandler` pattern above is the correct approach.

[VERIFIED: https://react-leaflet.js.org/docs/api-map/] ‚Äî `useMapEvents` hook signature and import path confirmed.

### Pattern 2: Migration 0003 ‚Äî Create find_photos, migrate existing rows, drop column

**What:** SQLite migration that creates `find_photos`, copies existing `finds.photo_path` values as primary photos, then removes the column.

**Key facts:**
- SQLite 3.35.0+ (released March 2021) supports `ALTER TABLE ... DROP COLUMN` [VERIFIED: https://www.sqlite.org/lang_altertable.html]
- The project uses `libsqlite3-sys = { version = ">=0.17.2", features = ["bundled"] }` which bundles a current SQLite version (3.46+ as of 2025) [ASSUMED ‚Äî confirm by checking bundled version at build time]
- `DROP COLUMN` fails if the column is a PRIMARY KEY, has a UNIQUE constraint, is indexed, or is in a CHECK constraint. `photo_path TEXT NOT NULL` has none of these ‚Äî it is safe to drop.
- The migration runner is `tauri-plugin-sql`'s built-in runner, which processes migrations in version order. Migration 0003 will run after 0002 on existing databases.

```sql
-- src-tauri/migrations/0003_find_photos.sql

-- 1. Create the new find_photos table
CREATE TABLE IF NOT EXISTS find_photos (
  id        INTEGER PRIMARY KEY AUTOINCREMENT,
  find_id   INTEGER NOT NULL REFERENCES finds(id) ON DELETE CASCADE,
  photo_path TEXT NOT NULL,
  is_primary INTEGER NOT NULL DEFAULT 0,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 2. Migrate existing finds.photo_path ‚Üí find_photos (is_primary = 1)
INSERT INTO find_photos (find_id, photo_path, is_primary, created_at)
SELECT id, photo_path, 1, created_at
FROM finds
WHERE photo_path IS NOT NULL AND photo_path != '';

-- 3. Create index for fast lookup by find_id
CREATE INDEX IF NOT EXISTS idx_find_photos_find_id ON find_photos(find_id);

-- 4. Drop the now-redundant column from finds
ALTER TABLE finds DROP COLUMN photo_path;

-- 5. Record schema version
INSERT OR REPLACE INTO app_metadata (key, value) VALUES ('schema_version', '3');
```

**lib.rs change:** Add Migration version 3 to the migrations vec and update the smoke test assertion from `assert_eq!(migrations.len(), 2)` to `assert_eq!(migrations.len(), 3)`.

### Pattern 3: delete_find Rust command

**What:** Rust command that deletes a find DB record (cascades to `find_photos`) and optionally moves all associated photo files to the system trash.

```rust
// Source: https://docs.rs/trash/latest/trash/ ‚Äî trash::delete() API
// Add to Cargo.toml: trash = "5"

use trash;
use rusqlite::{Connection, params};

#[tauri::command]
pub async fn delete_find(
    storage_path: String,
    find_id: i64,
    delete_files: bool,
) -> Result<(), String> {
    let conn = open_db(&storage_path).map_err(|e| e.to_string())?;

    if delete_files {
        // Collect photo paths before deleting DB rows
        let mut stmt = conn.prepare(
            "SELECT photo_path FROM find_photos WHERE find_id = ?1"
        ).map_err(|e| e.to_string())?;
        let paths: Vec<String> = stmt.query_map(params![find_id], |row| row.get(0))
            .map_err(|e| e.to_string())?
            .filter_map(|r| r.ok())
            .collect();

        // Move files to system trash (Recycle Bin on Windows)
        for rel_path in &paths {
            let abs_path = format!("{}/{}", storage_path, rel_path);
            // trash::delete returns Ok(()) even if file doesn't exist on some platforms
            if let Err(e) = trash::delete(&abs_path) {
                // Log but don't fail ‚Äî DB delete still proceeds
                eprintln!("trash::delete failed for {}: {}", abs_path, e);
            }
        }
    }

    // ON DELETE CASCADE handles find_photos rows automatically
    conn.execute("DELETE FROM finds WHERE id = ?1", params![find_id])
        .map_err(|e| format!("DB delete failed: {}", e))?;

    Ok(())
}
```

**trash crate decision:** `trash` v5.2.5 (the Rust crate, not the npm package) is the correct dependency. It uses the Windows Shell API (`SHFileOperation`/`IFileOperation`) for Recycle Bin ‚Äî not a raw file delete. No official Tauri plugin for trash exists. [VERIFIED: https://docs.rs/trash/latest/trash/] [VERIFIED: https://github.com/Byron/trash-rs]

### Pattern 4: Mode A import ‚Äî N photos ‚Üí 1 find

**What:** When Mode A is active, all selected photos map to a single `ImportPayload` group. The Rust `import_find` command needs to handle a group of source paths for one find record, inserting one `finds` row and N `find_photos` rows.

**How the Rust command changes:**
- `ImportPayload` gains an optional `additional_photos: Vec<String>` field (empty for Mode B)
- For Mode A calls: one `finds` row inserted, then one `find_photos` row per source path (first = `is_primary = 1`)
- For Mode B calls: existing one-find-one-photo behavior preserved

**Frontend state shape change:** `PendingItem` in ImportDialog needs to distinguish modes. Mode A groups photos under a single `ImportPayload` with `additional_photos`. Mode B keeps current shape (one item per photo). The `BulkMetadataBar` updates a shared "template" payload that gets applied to all items.

### Pattern 5: TanStack Query invalidation after schema change

**What:** The `Find` type in `finds.ts` currently has `photo_path: string`. After migration 0003, `photo_path` disappears from `finds` and lives in `find_photos`. The `Find` type and `get_finds` query must be updated to fetch photos separately or via a JOIN.

**Recommended approach:** Add a `photos: FindPhoto[]` field to `Find`. The `get_finds` Rust command fetches finds, then for each find fetches associated `find_photos` rows (or uses a single JOIN query). On the React side, `FindCard` receives `find.photos` and renders the strip ‚Äî no new query needed.

**Alternative:** A separate `useFindPhotos(findId)` query per card. This is n+1 and causes layout jank. Avoid it.

**Invalidation ‚Äî no change to pattern:** After any mutation (import, delete, update), call:
```typescript
qc.invalidateQueries({ queryKey: [FINDS_QUERY_KEY, storagePath] });
```
This refetches the full list including embedded photos. [VERIFIED: existing codebase pattern in `useUpdateFind` and `ImportDialog.tsx`]

### Anti-Patterns to Avoid

- **Calling `useMapEvents` outside a `MapContainer`:** This throws a context error. The hook must be in a child component rendered inside `MapContainer`. Use the `ClickHandler` sub-component pattern above.
- **Dropping `photo_path` before migrating data:** The INSERT into `find_photos` must precede the `DROP COLUMN`. Reversing the order loses all photo paths.
- **Using `std::fs::remove_file` for delete-with-disk:** This permanently deletes, bypassing the Recycle Bin. Always use `trash::delete()` when `delete_files = true`.
- **Fetching `find_photos` per-card with a separate query (n+1):** Embed photos in the `get_finds` response or use a JOIN.
- **Re-mounting `MapContainer` on every modal open:** `MapContainer` is not designed to be unmounted and remounted rapidly. Keep it mounted inside the Dialog and use the `key` prop only if you need to reset map state.

## Common Pitfalls

### Pitfall 1: MapContainer key/remount causing blank map
**What goes wrong:** If `LocationPickerMap` conditionally renders `MapContainer` based on `open` state (e.g. `{open && <MapContainer>}`), each modal open/close destroys and recreates the Leaflet map instance, causing a blank map or console errors about "map container is already initialized".
**Why it happens:** Leaflet attaches to a specific DOM element by ID. If that element is removed and re-added, Leaflet's internal state is out of sync.
**How to avoid:** Always keep `MapContainer` mounted inside the Dialog. Use `display: none` / CSS visibility rather than conditional rendering. Or keep the Dialog always mounted and control visibility via the `open` prop (Dialog from shadcn already handles this correctly ‚Äî the content is portaled but the component stays in the tree).
**Warning signs:** "Map container is already initialized" console error on second modal open.

### Pitfall 2: lib.rs migration smoke test will fail after adding Migration 3
**What goes wrong:** The existing `test_migration_vec_has_two_entries` test in `lib.rs` asserts `migrations.len() == 2`. Adding migration 3 without updating this test causes a CI failure.
**Why it happens:** The smoke test is a compile-time regression guard for the migration vec length.
**How to avoid:** Update the test to assert `migrations.len() == 3` and add `assert_eq!(migrations[2].version, 3)` in the same commit that adds the migration.
**Warning signs:** `cargo test` fails with "Must have exactly 2 migrations" assertion.

### Pitfall 3: `find_photos` ON DELETE CASCADE not firing in rusqlite
**What goes wrong:** `delete_find` deletes the `finds` row but `find_photos` rows remain because rusqlite has foreign key enforcement disabled by default.
**Why it happens:** SQLite foreign key cascade is opt-in per connection. `PRAGMA foreign_keys = ON` must be executed on the connection before any DML.
**How to avoid:** In `open_db()` (or in the `delete_find` command specifically), execute `conn.execute_batch("PRAGMA foreign_keys = ON;")?;` immediately after opening the connection. Check whether existing commands rely on FK enforcement too ‚Äî the current `import_find` does not use FKs, but it is safest to enable globally.
**Warning signs:** After delete, `SELECT COUNT(*) FROM find_photos WHERE find_id = ?` returns non-zero.

### Pitfall 4: `photo_path` still referenced in existing Rust code after DROP COLUMN
**What goes wrong:** `get_finds`, `import_find`, and `update_find` all SELECT or INSERT `photo_path`. After migration 0003 drops the column, these queries fail at runtime.
**Why it happens:** SQLite returns an error if a named column in a SELECT/INSERT doesn't exist. The Rust `FindRecord` struct still has `photo_path: String`.
**How to avoid:** In the same plan wave that adds migration 0003, update ALL Rust commands: remove `photo_path` from `FindRecord`, update SELECT statements to exclude it, update INSERT statements to exclude it, and add `get_find_photos` command.
**Warning signs:** "table finds has no column named photo_path" rusqlite error at runtime.

### Pitfall 5: `useMapEvents` called outside `MapContainer`
**What goes wrong:** `useMapEvents` throws "No map context available. Did you add your component inside a MapContainer or Map component?" if called outside a MapContainer.
**Why it happens:** `useMapEvents` uses React context provided by `MapContainer`. If the component tree is not nested inside `MapContainer`, the context is undefined.
**How to avoid:** Always put `useMapEvents` in a separate child component (like the `ClickHandler` pattern shown above) that renders inside `MapContainer`.
**Warning signs:** "No map context available" error on first render.

### Pitfall 6: Mode A photo ordering in FindPreviewCard
**What goes wrong:** In Mode A, multiple photos are shown in a single preview "card". If the UI uses the same `FindPreviewCard` component with a single `sourcePath`, the thumbnail only shows one photo and there's no way to reorder or designate the primary photo.
**Why it happens:** Current `FindPreviewCard` is designed for one photo ‚Üí one find.
**How to avoid:** For Mode A, either (a) use a different preview component that accepts `sourcePaths: string[]` and shows a mini strip, or (b) show only the first photo as the thumbnail and indicate "N photos". The simpler option is (b) ‚Äî display count badge.

## Code Examples

### get_find_photos Rust command
```rust
// Source: established rusqlite pattern from existing import.rs
#[derive(serde::Serialize)]
pub struct FindPhoto {
    pub id: i64,
    pub find_id: i64,
    pub photo_path: String,
    pub is_primary: bool,
}

#[tauri::command]
pub async fn get_find_photos(
    storage_path: String,
    find_id: i64,
) -> Result<Vec<FindPhoto>, String> {
    let conn = open_db(&storage_path)?;
    conn.execute_batch("PRAGMA foreign_keys = ON;").map_err(|e| e.to_string())?;
    let mut stmt = conn.prepare(
        "SELECT id, find_id, photo_path, is_primary FROM find_photos WHERE find_id = ?1 ORDER BY is_primary DESC, id ASC"
    ).map_err(|e| e.to_string())?;
    let photos = stmt.query_map(params![find_id], |row| {
        Ok(FindPhoto {
            id: row.get(0)?,
            find_id: row.get(1)?,
            photo_path: row.get(2)?,
            is_primary: row.get::<_, i64>(3)? == 1,
        })
    })
    .map_err(|e| e.to_string())?
    .collect::<Result<Vec<_>, _>>()
    .map_err(|e| e.to_string())?;
    Ok(photos)
}
```

**A4 detail:** shadcn/ui `Dialog` with `DialogContent` uses Radix UI `Dialog.Content`, which by default unmounts children when the dialog is closed (`forceMount` defaults to false). This WILL cause the "map container is already initialized" error on second open. Solution: pass `forceMount` to `DialogContent` and use CSS to hide it when closed, OR destroy+recreate with a stable key. The simplest correct approach is to use the Dialog's `open` prop to show/hide without unmounting:

```typescript
// In LocationPickerMap: use forceMount to prevent unmount/remount of MapContainer
<DialogContent forceMount style={{ display: open ? undefined : 'none' }}>
```

Or use a `key` prop on `MapContainer` that only changes when truly needed (e.g., `key={open ? 'open' : 'closed'}` forces recreation each open, which is acceptable for a picker modal that rarely opens).

## Open Questions

1. **get_finds JOIN strategy ‚Äî two queries vs. JSON aggregation**
   - What we know: rusqlite supports both approaches. Two queries + HashMap assembly in Rust is straightforward. SQLite JSON aggregation (`json_group_array`) is available but adds complexity.
   - What's unclear: Performance difference for typical collections (10‚Äì500 finds).
   - Recommendation: Use two separate queries + HashMap assembly. Simpler, testable, and the dataset is small enough that two queries are negligible.

2. **Mode A ‚Äî how does the Rust command receive multiple source paths for one find?**
   - What we know: Current `ImportPayload` has one `source_path`. Mode A requires N paths ‚Üí 1 find.
   - What's unclear: Whether to change `ImportPayload` to `source_paths: Vec<String>` (unified shape) or send the first as `source_path` and additional as `additional_photos: Vec<String>`.
   - Recommendation: Use `additional_photos: Vec<String>` (additive change, backward compatible with Mode B which sends empty vec).

3. **Post-import review dialog ‚Äî how to pass just-imported finds to it?**
   - What we know: `ImportSummary.imported` already returns the full `FindRecord` list of what was just imported. The review dialog needs these records.
   - What's unclear: Should the review dialog fetch fresh records from DB or use the `imported` slice from the summary?
   - Recommendation: Use `summary.imported` directly ‚Äî they're already accurate and this avoids an extra DB round-trip. But since `Find` type is changing to include `photos: FindPhoto[]`, `ImportSummary.imported` must also return embedded photos.
