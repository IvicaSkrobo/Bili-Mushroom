# Phase 06-05 Plan: Map Scalability

## Goal

Keep the map responsive by rendering aggregate/visible data instead of mounting every find as a Leaflet marker.

## Scope

- Viewport/zoom-aware map query path.
- Cluster/group summaries by visible bounds.
- Drill-down from cluster/location/species to collection/detail.
- Fit-to-pins behavior that can use bounds from SQLite instead of all records in memory.

## Tasks

1. Define map summary commands.
   - Visible bounds query: north/south/east/west + zoom/filter.
   - Group by exact coordinate or grid bucket depending on zoom.
   - Return counts, species samples, representative photo path, and aggregate bounds.

2. Replace all-client grouping for large paths.
   - Keep current `CollectionPins` behavior for small result sets if useful.
   - Use summary markers/clusters when result count exceeds a threshold.

3. Keep map workflows intact.
   - Species filter.
   - Zone drawing/editing.
   - Fit-to-pins.
   - Open collection/species from popup.

4. Handle dense/crowded locations.
   - Cluster count marker at low zoom.
   - Location/species popup only loads details when opened.
   - Avoid O(n^2) label overlap work for huge marker counts.

## Success Criteria

- Map interactions do not require all finds loaded in frontend memory.
- Large libraries render a bounded number of markers/clusters per viewport.
- Fit-to-pins works through aggregate bounds.
- Existing zone editing remains usable.

## Validation

- Unit-test grouping/bounds query behavior.
- Manual map smoke test with generated 10k+ coordinate records.
- Verify zoom/pan/refilter does not create runaway marker counts.
