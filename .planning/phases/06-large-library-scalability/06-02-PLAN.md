# Phase 06-02 Plan: Database and Query Shapes

## Goal

Stop normal app browsing from loading every find and every photo row when the user only needs summaries.

## Scope

- New lightweight query commands beside existing full-record commands.
- Paginated find/species summaries.
- On-demand photo/detail queries.
- SQLite index audit for date, species, location, and photo lookup paths.

## Tasks

1. Add summary models.
   - Species summary: species name, find count, photo count, latest date, favorite count, representative photo id/path.
   - Find summary: id, species, date/location fields, first/primary photo only, counts.
   - Full find detail remains available on demand.

2. Add paginated commands.
   - List species summaries with search/filter.
   - List finds for a species/date filter with limit/offset or cursor.
   - Fetch photos for a find only when expanded/lightbox/edit needs them.

3. Audit and add indexes.
   - `finds(species_name)`.
   - Date/species compound indexes if query plans need them.
   - `find_photos(find_id, is_primary, id)`.
   - Optional `find_photos(photo_path)` for duplicate checks if missing.

4. Update frontend hooks incrementally.
   - Keep `useFinds` for legacy surfaces during migration.
   - Add summary/detail hooks and targeted invalidation.

## Success Criteria

- Collection and map can render from summary/group data without full photo arrays for every find.
- Detail/lightbox/edit flows still load complete data when needed.
- Large-library queries have bounded result sizes.

## Validation

- Add Rust query tests for pagination and summary counts.
- Use `EXPLAIN QUERY PLAN` during development to verify indexes are used.
- Add frontend hook tests around pagination/cache invalidation.
