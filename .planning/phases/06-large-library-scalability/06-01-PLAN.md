# Phase 06-01 Plan: Library Folder Semantics and Large Import/Relocation

## Goal

Make library folder selection unambiguous, then make folder/library import safe for multi-thousand-photo source folders without rendering or holding the full selection as UI thumbnails. This includes the case where a user wants to copy or move an existing photo folder into a different Gljivobook library location.

## Scope

- Batch folder scanning and import progress.
- Bounded in-memory queue of candidate photos.
- Duplicate detection that can skip already-registered photo paths.
- Cancel/resume-friendly checkpoints where practical.
- UX that shows counts, current file, and representative previews only.
- Explicit folder handling modes:
  - Open existing Gljivobook library folder with an existing database.
  - Create/switch to an empty Gljivobook library folder.
  - Register/use selected folder in place.
  - Copy selected folder/photos into the current library.
  - Move selected folder/photos into the current library with copy-before-delete safety.
- Tooltips/help text for every ambiguous folder action.

## Tasks

1. Audit current folder chooser paths.
   - First-run folder picker.
   - Settings Change Folder.
   - Storage path persistence.
   - Current auto-scan behavior after path changes.

2. Add explicit chooser semantics.
   - Detect whether selected folder already has a Gljivobook database.
   - Detect whether selected folder appears empty.
   - Detect whether selected folder appears to contain photos/subfolders but no Gljivobook database.
   - Ask the user which action they mean instead of silently treating all folders the same.

3. Add tooltips and inline help.
   - Existing library: explain that the app will open that database and not copy files.
   - Empty folder: explain that a new Gljivobook database will be created there.
   - Register in place: explain that original photos stay where they are and Gljivobook records them there.
   - Copy into library: explain that originals remain and copies are created in the Gljivobook folder.
   - Move into library: explain copy-before-delete safety and that originals are removed only after successful copy.
   - Destructive or long-running options should also show confirmation text before starting.

4. Audit current import paths.
   - `ImportDialog` folder picker and `scanAndImport`.
   - Rust `import_find` copy/register loop.
   - Existing duplicate detection in `find_photos`.

5. Add a batch import shape.
   - Scan folders in chunks.
   - Keep memory bounded.
   - Emit progress by scanned/imported/skipped counts.
   - Avoid building one giant `photos` array for large imports.

6. Add safe folder relocation modes.
   - Do not imply that changing the library path moves existing data unless it really does.
   - Copy mode preserves the original source folder.
   - Move mode copies into the library first, verifies destination, then deletes source files/folders only after successful copy.
   - Report partial failures without losing successfully copied data.

7. Replace large thumbnail rendering.
   - Show count and a small sample preview.
   - Keep single-find multi-photo import ergonomic for normal small batches.
   - Add warning/copy for very large batches without blocking them.

8. Preserve local-first safety.
   - Copy before delete.
   - Keep delete-source explicit.
   - Record delete failures without aborting successful imports.

## Success Criteria

- Choosing an existing Gljivobook folder opens that library without re-copying photos.
- Choosing an empty folder clearly creates/switches to a new empty library.
- Choosing a photo folder without a Gljivobook DB asks whether to register, copy, or move.
- Every folder action has a tooltip or inline help text that states whether originals remain, whether files are copied, and where the database is stored.
- Importing a folder with thousands of photos does not mount thousands of image tags.
- User can choose register-in-place, copy, or move semantics and the app copy/move behavior matches that label.
- Progress continues to update during long import.
- Skipped duplicates are counted and reported.
- Existing small import workflow still feels simple.

## Validation

- Add a synthetic scan/import test fixture.
- Run manual import with a large generated photo-path set or lightweight test images.
- Test copy and move modes with nested folders and simulated locked-file/delete failures.
- Verify memory does not grow with the full candidate count in the import UI.
