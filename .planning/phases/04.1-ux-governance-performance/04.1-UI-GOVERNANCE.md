# 04.1 UI Governance Baseline (Nocturne Herbarium)

Status: Active baseline
Created: 2026-04-16
Owner: Phase 04.1

## 1) Goal

Prevent visual drift after the identity refresh by enforcing one canonical system for tokens, component variants, motion, and QA.

## 2) Canonical Sources

- Global tokens and interaction baseline: `src/index.css`
- App shell chrome behavior: `src/components/layout/AppShell.tsx`
- Stats visual hierarchy reference: `src/tabs/StatsTab.tsx`

All new UI work must align to these before introducing new styles.

## 3) Token Policy

### 3.1 Color tokens
- Use semantic tokens only (`--background`, `--foreground`, `--card`, `--primary`, etc.).
- Do not hardcode one-off hex/oklch values inside component classes unless in documented exception cases (e.g., map marker compatibility constraints).
- New tokens require adding both light and dark values.

### 3.2 Typography tokens
- Serif display: `--font-serif`
- UI/body: `--font-sans`
- Technical/meta text: `--font-mono`

Do not introduce additional font families without explicit design review approval.

### 3.3 Spacing/radius
- Prefer existing spacing scale in component classes.
- Keep `--radius` as global roundness source; avoid arbitrary rounded values in isolated components.

## 4) Component Variant Policy

### 4.1 Buttons / inputs / tabs
- Reuse existing UI primitives in `src/components/ui/*`.
- New variants must be added centrally (primitive file) rather than ad-hoc class stacks across pages.

### 4.2 Shell surfaces
- Header/tab surfaces should keep translucent-card + border layering pattern.
- If adjusting shell contrast, update both AppShell and token layer together.

### 4.3 Stats surfaces
- Stats heading and export bar are reference patterns for hierarchy and utility actions.
- New stats cards/blocks should follow current spacing rhythm before introducing new layout systems.

## 5) Motion & Accessibility Policy

### 5.1 Motion
- Use existing motion primitives (`animate-fade-up`, `stagger-item`) first.
- Keep durations in the current subtle range; avoid introducing rapid or distracting animation.

### 5.2 Accessibility
- Maintain visible focus states on interactive controls.
- Preserve readable contrast in both themes for text and icon-only actions.
- Avoid reducing hit targets below existing icon-button sizing.

## 6) Anti-Drift Rules (Hard Constraints)

1. No new global colors/fonts without token update + rationale.
2. No page-local redefinition of shell typography hierarchy.
3. No direct styling bypass of primitives when a variant extension is possible.
4. Any exception must be logged in plan summary under "Deviations".

## 7) PR Review Checklist (UI changes)

- [ ] Uses existing semantic tokens only (or token additions documented)
- [ ] Uses existing primitives/variants where possible
- [ ] Light + dark validated
- [ ] Focus/hover/disabled states checked
- [ ] Motion remains subtle and consistent
- [ ] Screenshot evidence attached for perceptible UI changes
- [ ] Planning docs updated if conventions changed

## 8) Handoff to Next Plans

- 04.1-02 should enforce chunk/perf guardrails without violating this visual baseline.
- 04.1-03 E2E should include UI assertions for key shell/stats states.
- 04.2-x (insights/hints) must adopt this baseline rather than introducing parallel visual language.
